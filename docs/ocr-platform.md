# Receipt OCR platform notes

OpenExpense receipt scanning is a browser-only pipeline. It should stay private,
fast enough for mobile phones, and readable for maintainers who need to tune OCR
or parsing without learning the whole app first.

## Pipeline tags

`src/features/receipt.js` exposes `Receipt.PIPELINE_TAGS` as human-readable code
tags for the major receipt stages:

| Tag | Stage |
| --- | --- |
| `ocr.engine.lazy-cdn` | Lazy load `ppu-paddle-ocr` only when the user scans or the idle warm-up runs. |
| `ocr.model.browser-cache` | Download OCR models on first use, then rely on the browser cache. |
| `ocr.pdf.text-first` | Extract embedded PDF text before falling back to OCR. |
| `ocr.pdf.page-one-preview` | Render page 1 as the review thumbnail and OCR fallback canvas. |
| `ocr.image.canvas-normalize` | Resize photos/PDF renders into the OCR-friendly canvas range. |
| `ocr.review.human-confirm` | Require user review before anything is written to the ledger. |

Use these tags in comments, diagnostics, or future tests when referring to a
specific stage. They are not analytics events and must not send data anywhere.

## Cross-platform behavior

| Platform capability | Where it lives | Behavior |
| --- | --- | --- |
| Camera-first capture | `Utils.prefersCamera()` and `Receipt.pickImage()` | Adds `capture="environment"` on coarse-pointer/mobile-size devices so phones can open the rear camera. |
| File upload fallback | `index.html` file input | Accepts images, HEIC/HEIF, and PDF files from every browser that supports file inputs. |
| Desktop save picker | `Utils.canUseSavePicker()` and `features/ledger.js` | Uses File System Access API on secure, non-mobile contexts when available. |
| Mobile sharing | `features/ledger.js` | Uses Web Share with files when the platform supports it, then falls back to download links. |
| PWA shell | `manifest.json`, viewport meta, CSS safe areas | Supports installable mobile/desktop browser use without a native wrapper. |

## Performance guardrails

- Keep OCR and PDF readers lazy-loaded from CDN. The initial app bundle should not
  include OCR models or PDF.js.
- Keep photo and PDF canvases inside the named OCR side range in
  `receipt.js` (`OCR_IMAGE_MIN_SIDE` to `OCR_IMAGE_MAX_SIDE`). This preserves
  text detail while avoiding very large canvases on mobile CPUs and GPUs.
- Prefer embedded PDF text before OCR. Text extraction is faster, avoids model
  work, and preserves accuracy for invoices generated by desktop software.
- Keep the review thumbnail JPEG quality explicit. It should be clear when a
  future change trades memory size for visual fidelity.
- Warm the OCR engine during idle time only. User input and first render should
  remain the priority on mobile browsers.

## Upgrade checklist

When updating OCR, ONNX, PDF.js, or platform APIs:

1. Update the CDN pin in `src/features/receipt.js`.
2. Update any peer-dependency import map pins in `index.html`.
3. Re-run `npm run build` and commit the generated `app.js` and current chunks.
4. Smoke test at least one mobile browser and one desktop browser:
   - photo receipt from the camera or photo picker,
   - PDF with embedded text,
   - scan-only PDF or screenshot fallback,
   - export/share/download fallback.
5. Confirm all extracted fields still land in the review sheet before save.

OpenExpense should not add cloud OCR, background uploads, or server-side parsing
without an explicit product decision. The privacy boundary is that receipt
images, raw OCR text, and ledger data stay on the user's device.
